# M3 Self-Host Bootstrap — Cursor execution prompt

## Problem

yoyo self-hosting compiler: `gen2→gen3` crashes with `0xC0000005`.

Current confirmed state:
- **gen1→gen2**: OK (exit 0). gen2 handler map has **132/256 valid entries** (H_00, H_01, H_0c..H_xx).
- **gen2→gen3**: CRASH — gen2 runs but its emitted x64 code is buggy.

Root cause: gen2's `.text` handlers are **scan-emitted** (via blob 2's H_30 meta-emitter) with **wrong x64 bytes** — wrong fixup relocations, wrong state variable offsets, or wrong call targets. Node's `compileFromAnalyzed` is the golden reference.

Handler map (PE `data+0xFE00`, file offset `0x18600`):
```
H_00 = 0x54, H_01 = 0x8ff, H_0c = 0x1a0f, ... (132 entries)
codeEnd = 0x5baa (23466)
```

### The two codegen paths

```
yoyo.ty
  ├── Node (yoyo.js)        → compileFromAnalyzed() → correct x64 (via JS emit-x64.js)
  └── gen1 (yoyo.exe)       → scan-emit blobs       → buggy x64 (via blob 2 H_30 emitter)
                              (blobs generated by yoyo-gen.js using encode-x64.js)
```

Both read the same yoyo.ty. Both produce the same handlers. But the x64 bytes differ. Node's path is correct; the scan-emit path (embedded in yoyo.ty's `data.blob` lines) has bugs.

### The 4 blobs

```
Blob | Offset  | Size    | Purpose                   | Source in yoyo-gen.js
─────|─────────|─────────|───────────────────────────|─────────────────────────────
0    | 0x4000  | 124928B | Main runtime + codeEnd    | buildStartup()
1    | 0xcc00  |    84B  | Entry stub                | buildStartup() (entry part)
2    | 0xcc54  | 23382B  | H_30 meta-emitter + scan  | buildWinHandlerOverlay()
3    | 0x14c54 |  1028B  | H_E0-H_EC fixup logic     | buildWinHandlerOverlay()
```

## Concrete diff evidence (gen2 vs Node reference)

H_00 comparison (2219 bytes gen2 vs 6889 bytes Node):

1. **`ff15` / `e8` rel32 values differ systematically** — call_rip to IAT and call to handlers have different rel32 because gen2 handlers are at different .text offsets. HOWEVER, the **target addresses must be correct**. If gen2's fixup resolver (H_E0-H_EC in blob 3) computes wrong targets, calls go to wrong addresses → crash.

2. **`memcpy.state` uses different register allocation** — gen2 emits `mov rsi, [r8+0x40]` / `mov rdi, [r8+0x60]` pattern for `memcpy.state` ops. If RSI/RDI are live across the memcpy, they get clobbered.

3. **Last 5 calls (`call H_e0`, `call H_e5`) use wrong rel32** — gen2 computes `e8 3d 13 00 00` (target = current+5+0x133d) but Node computes `e8 9a 46 00 00` (target = current+5+0x469a). Only ONE of these is correct for reaching the actual H_e0 handler. The gen2 value may point to wrong memory → 0xC0000005.

**Key diagnostic**: compare the actual call targets:
```
gen2: call H_e0 → instruction at offset X, rel32=0x133d → target=0x1c2d
Node: call H_e0 → instruction at offset Y, rel32=0x469a → target=0x???
```
In Node reference, `extractWinHandlerSlices` reports where H_e0 actually is. The gen2 call should reach the same place.

### Step 2 — Read source files and understand architecture

Read these 4 files in order:

1. `src/yoyo-gen.js` — Source of all 4 blobs. Search `buildStartup()` and `buildWinHandlerOverlay()`. These JS functions use encode-x64.js to produce blob bytes. **This is the source of truth**, NOT the hex in yoyo.ty.

2. `src/encode-x64.js` — x64 instruction helpers (`E.mov_ri`, `E.call_rip`, `E.lea_rip`, `E.jmp_rel`, `E.jcc32`). Used by BOTH yoyo-gen.js (blob generation) AND backends (compileFromAnalyzed, golden reference).

3. `src/backends/win-emit-core.js` — Node's `compileFromAnalyzed`. The **golden reference** emit path. See how it emits x64 for each TIR op.

4. `src/win-handler-overlay.js` — Builds the handler image embedded as blob 2 in yoyo.ty.

### Step 4 — Trace diff to blob code

For a handler that differs (e.g. H_00), find it in yoyo.ty source, then find which blob function in yoyo-gen.js produces the code that emits it. The scan loop in blob 2 calls H_30 for each handler; trace H_30's logic to find the emission bug.

### Step 5 — Apply fix

**Choice A** (faster, 3-5 days): Fix the scan-emit bug in yoyo-gen.js blob generation. Find the specific wrong offset/relocation/opcode in blob 2, fix the JS source, regenerate yoyo.ty, rebuild gen1, retest.

**Choice B** (slower, 2+ weeks): Replace `data.blob` with TIR intrinsics. Start with blob 3 (1028 bytes, simplest), then blob 1 (84B), blob 2 (23KB), blob 0 (125KB). Each replacement verified: `node scripts/compare-backends.js --target=win` exits 0.

### x64 → TIR translation

x64 bytes → TIR form:

```
48 B8 <8B>    → state.set 0 <val>       # mov rax, imm64
48 BA <8B>    → state.set 2 <val>       # mov rdx, imm64
49 B8 <8B>    → state.set 8 <val>       # mov r8, imm64
E8 <4B rel>   → call H_<target>
E9 <4B rel>   → jmp H_<target>
0F 8C/8F/84  → jcc.lt / jcc.gt / jcc.eq
C3            → ret
FF 15 <4B>    → call [IAT]              # call_rip (import)
90            → (skip, NOP padding)
```
