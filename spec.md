# mini-kyc Spec — Self-Hosting Compiler on Windows x64

## 0. TL;DR

`mini-kyc` is a self-hosting compiler that takes `.ky` source text (a custom bytecode)
and produces native Windows x86_64 `.exe` files. The compiled output runs without any
runtime dependencies — no DLL, no CRT, no Node, no Python. The compiler itself is
written in `.ky` and can compile its own source.

The project is bootstrapped in three stages. This document describes the architecture,
invariants, and development workflow so future contributors (human or AI) can make
changes without re-discovering the landmines.

---

## 1. Definition of Self-Hosting

"Self-hosting" means three increasingly strict invariants:

| Stage | Property | How to verify |
|-------|----------|---------------|
| **Stage 1 (done)** | Generator + compiler are bit-deterministic | `bash scripts/bootstrap-check.sh --strict --lock` compares SHA256 against `bootstrap-baseline.txt` |
| **Stage 2 (next)** | Self-hosted compiler produces same output as host compiler | `mini-kyc.exe projects/mini-kyc.ky output-self.exe` should `cmp` equal to `ky-compiler.js`'s output |
| **Stage 3 (after)** | Three-stage bootstrap converges | `mini-kyc2.exe` (Stage 2 output) compiling `mini-kyc.ky` produces `mini-kyc3.exe` that is bit-identical to `mini-kyc2.exe` |

Without Stage 1, you can't tell whether your change is a regression. Without Stage 2,
you don't have a compiler. Stage 3 is the formal proof of self-hosting — but it's
useless if Stage 1 is broken.

```
┌────────────────────────────────┐

│  (host compiler: Node.js)      │
│  - parse .ky tokens            │
│  - dispatch by opcode          │
│  - emit x86_64 into code buf   │
│  - resolve fixups              │
│  - call pe-builder.build()     │
└──────────────┬─────────────────┘
               │ writes

               │
```

The generator and host compiler are **build-time tools**. Only `mini-kyc.exe` runs at
user time. Stage 2 requires that the self-hosted compiler (running inside `mini-kyc.exe`)
produce bit-identical output to what `ky-compiler.js` produces.

## 4. PE Layout (Every Output Program)

All output `.exe` files (including `mini-kyc.exe` itself) share this layout:

```
RVA       File offset   Content
─────────────────────────────────────────────────
0x0000    0x0000        DOS header (PE signature at 0xF0)
0x0400    0x0400        .text section (CODE, R+X)
                         - entry point at offset 0
                         - startup blob at offset 0 (68 bytes)
                         - emitted handler code after startup
                         - handler table, fixup arrays
0x5000    0x4400        .rdata section (R)
                         - IAT (8 entries × 8 bytes)
                         - import table
                         - data blobs (PE template, startup blob,
                           strings table, etc.)
0x8000+                 (empty / padding)
```

Layout constants (must agree between generator and host compiler):

```javascript
// create-mini-kyc3.js and ky-compiler.js
const TEXT_VS = 0x4000;   // .text virtual size
const CODE_RVA = 0x1000;  // .text base RVA
const IAT_BASE = 0x5000;  // IAT base (= CODE_RVA + TEXT_VS)
```

### IAT Entry Layout

| Index | Function | IAT RVA |
|-------|----------|---------|
| 0 | ExitProcess | 0x5000 |
| 1 | GetStdHandle | 0x5008 |
| 2 | WriteFile | 0x5010 |
| 3 | ReadFile | 0x5018 |
| 4 | CreateFileA | 0x5020 |
| 5 | GetFileSize | 0x5028 |
| 6 | CloseHandle | 0x5030 |
| 7 | VirtualAlloc | 0x5038 |

The order is fixed — `FUNCS` array in `create-mini-kyc3.js:8` and the `addImport` call
in `ky-compiler.js:48` / `pe-builder.js` must agree.

### Embedded PE Template

The generator builds a 0x4000-byte PE template at runtime (`tplPE.build()` in
`create-mini-kyc3.js:25`) and embeds it in the compiler's data section at offset
`PE_BLOB_OFF = 0x4000`. This template is what gets copied to the output buffer when
the self-hosted compiler processes a `84` opcode (memcpy).

**Currently `84` is in Phase 2 (not yet supported).** Phase 1 outputs only the 0x8800
prefix of the template — the rest is zeros. See §14 for current limitations.

### Startup Blob (68 bytes, at data offset `STARTUP_BLOB_OFF = 0xCC00`)

This is the x86_64 code emitted at offset 0 of every output program's `.text` section.
It's a static, hand-built blob in `buildStartup()`:

```asm
movabs rcx, 0          ; lpAddress = NULL
movabs rdx, 0x20000    ; dwSize = 128KB
movabs r8, 0x3000      ; MEM_COMMIT | MEM_RESERVE
movabs r9, 0x40        ; PAGE_EXECUTE_READWRITE
call [rip + VirtualAlloc_IAT_disp]  ; → RAX = state array base
mov  r15, rax          ; R15 = state array base
movabs rcx, -11        ; STD_OUTPUT_HANDLE
call [rip + GetStdHandle_IAT_disp]  ; → RAX = stdout handle
mov  r14, rax          ; R14 = stdout
ret                    ; returns to caller (main top-level)
```

This blob's RIP-relative IAT displacements are **computed at generator time** using:

```javascript
disp = IAT[func] - (CODE_RVA + offset_within_blob + 6)
```

These displacements are baked into the blob and never change at runtime. The same
formula is used for the embedded startup blob in `mini-kyc.exe` and for any IAT call
emitted by `H_60`/`H_61`/`H_62` handlers (Phase 2).

**Known bug (dormant until Phase 2)**: see `FLASH-V4-REVIEW.md` §A.1. The IAT disp
constants in some emitter code were off by 2. The fix is in `.flash-v4-create-mini-kyc3.patch`
but should be re-derived, not blindly applied, before Phase 2 work starts.

## 5. Runtime State (Inside `mini-kyc.exe`)

After the startup blob runs, the runtime has:

| Register | Meaning | Set by |
|----------|---------|--------|
| `R15` | State array base (allocated by VirtualAlloc) | startup |
| `R14` | Stdout handle | startup |
| `R12` | Last file size (from GetFileSize) | handler |
| `R13` | Last file handle (from CreateFileA) | handler |
| `R8`, `R9`, `RCX`, `RDX`, `RSI`, `RDI` | Scratch — may be clobbered by any handler | handlers |

The state array is 0x20000 bytes = 16384 slots × 8 bytes. Slots are accessed via
`mov rax, [r15 + slot*8]` (load) or `mov [r15 + slot*8], rax` (store).

### Reserved State Slots (convention)

| Slot | Name | Purpose |
|------|------|---------|
| 0x00 | `state_00` | scratch |
| 0x02 | `state_02` | output buffer (allocated by `20 02 ...` at startup) |
| 0x03 | `state_03` | write_base = state_02 + 0x400 (start of `.text` in output) |
| 0x04 | `state_04` | handler offset table base |
| 0x05 | `state_05` | fixup_hh array base |
| 0x06 | `state_06` | fixup_pos array base |
| 0x07 | `state_07` | fixup_count |
| 0x09 | `state_09` | first_handler_flag |
| 0x0A | `state_0A` | input file buffer pointer (set by `50 0A 00` LoadFile) |
| 0x0B | `state_0B` | input file size (set by LoadFile) |
| 0x0C | `state_0C` | scanner read_ptr |
| 0x0D | `state_0D` | scanner end_ptr (= read_ptr + file_size) |
| 0x0E | `state_0E` | scanner code_offset (write position in `.text` section) |
| 0x0F | `state_0F` | scanner byte accumulator / opcode |
| 0x10 | `state_10` | scanner state (0..5) |
| 0x11 | `state_11` | scanner accumulator (hex digit) |
| 0x12 | `state_12` | scanner digit_count |
| 0x13 | `state_13` | scanner opcode |
| 0x14 | `state_14` | scanner arg_index |
| 0x40..0x49 | `state_40..49` | scratch for handlers |
| 0x50..0x52 | `state_50..52` | current opcode arguments |
| 0x45 | `state_45` | byte-to-emit (via `H_E0`) |

## 7. `.ky` Text Format

Plain ASCII, one instruction per line, space-separated hex tokens, `;` starts line comment.

```
40 23              ; handler start, label H_23
30 50 00           ; state_50 = 0
30 51 03           ; state_51 = 3
66 50              ; state_50++
65 50 51           ; cmp state_50, state_51
71 24              ; je H_24 (backward jump — known offset)
70 23              ; jmp H_23 (loop back)
40 24
FF                 ; ret
```

### Tokenization rules

- Tokens are hex digits (`0-9a-fA-F`), 1-2 chars each
- Space or newline separates tokens
- `;` to end of line is comment
- Strings (opcode 0x12) use `s<text>` form: `12 00 sHello, world!`
- Raw hex blobs (opcode 0x13) use raw hex: `13 cc00 s48b90000000000000000...`

### Scanner states

| State | Action on byte |
|-------|----------------|
| 0 (looking for opcode) | skip whitespace; `;` → comment; hex → state 1 |
| 1 (accumulating opcode) | hex → continue; ws → finalize, dispatch |
| 2 (looking for arg) | skip whitespace; `;` → comment; hex → state 3 |
| 3 (accumulating arg) | hex → continue; ws → finalize, dispatch when args complete |
| 4 (string mode) | collect until terminator |
| 5 (comment) | skip until `\n` or `\r` |

When all args for an opcode are accumulated, the scanner dispatches to the handler chain
(`H_30 → H_3A-H_3F → H_4A → H_4B → H_4C`). Each dispatch slot checks one opcode and
either calls the corresponding emitter (`H_31`-`H_39`, `H_40`-`H_44`) or falls through.

---

## 8. Handler Table Layout

256 entries × 4 bytes = 1024 bytes, allocated by `20 04 400` at startup.
Each entry is a u32 LE: the byte offset within the output's `.text` section where that
handler's code begins.

When the scanner dispatches to opcode X, it looks up `handler_table[X]`, gets the offset,
adds the offset to `write_base`, and `call`s there.

Forward references (calls to handlers not yet emitted) are recorded in the fixup arrays
and patched at EOF by the fixup resolver (`H_63`/`H_64`/`H_65`).

---

## 9. Fixup Mechanism

Forward jumps/calls can't be resolved at emit time (the target handler hasn't been emitted
yet). The emitter records:

- `state_05`: fixup_hh array (max 256 entries × 4 bytes) — handler index per fixup
- `state_06`: fixup_pos array — code_offset of the `disp32` field per fixup
- `state_07`: fixup_count

At EOF (when `read_ptr >= end_ptr`), the scanner JAE to `H_62`. `H_62`:
1. Emits a `C3` (RET) to close the last handler
2. Calls `H_63` which iterates the fixup arrays:
   - reads `hh` from fixup_hh[i]
   - reads `patch_pos` from fixup_pos[i]
   - looks up `handler_table[hh]` = target offset
   - computes `rel32 = target - (patch_pos + 4)`
   - writes `rel32` as u32 at `write_base + patch_pos`
3. Returns to the caller (top-level code)

**Current bug (must fix for Stage 2)**: The `H_62` label is overloaded — `JAE(0x62)`
in the scanner jumps there at EOF, AND `opcode 0x51` (WriteFile) also dispatches to
`H_62` as its emitter. The ky-compiler's analyze step overwrites the first H_62 with
the second. See `FLASH-V4-REVIEW.md` §B.2 for the proposed fix (rename EOF handler
to `H_1E`).

## 10. Generator Structure (`create-mini-kyc3.js`)

```javascript
// Helpers (lines 53-86) — text output
const hx = (n, w) => n.toString(16).padStart(w || 2, '0');
const L  = s => lines.push(s);        // emit a code line
const C  = s => lines.push('; ' + s); // emit a comment
const B  = ()  => lines.push('');     // blank line

// Opcode macro shorthand
const SET  = (s, v)    => '30 ' + hx(s) + ' ' + hx(v);
const GET  = (d, s)    => '60 ' + hx(d) + ' ' + hx(s);
const ADD  = (s, v)    => '61 ' + hx(s) + ' ' + hx(v);
const ADDV = (a, b)    => '68 ' + hx(a) + ' ' + hx(b);
const SUBV = (a, b)    => '69 ' + hx(a) + ' ' + hx(b);
// ...

// Handler macro
const H = (h) => '40 ' + hx(h, 2);

// Call handler macro (sets up a fixup)
const CH = (h) => '41 ' + hx(h, 2);
```

**Three concerns, three sections**:

1. **Top-level (lines 90-165)** — runs once at program startup. Sets up state array,
   loads input file, copies PE template, allocates tables, copies startup blob, calls
   scanner, writes output.

2. **Scanner (lines 168-540)** — the main loop. Reads input bytes, dispatches by
   opcode to handler. EOF handler at `H_62` (rename to `H_1E` is the proposed fix).

3. **Handlers (lines 600-853)** — one section per handler. Each handler is a sequence
   of `L()` calls that emit KY instructions, which ky-compiler.js compiles into x86_64.

**Adding a new opcode emitter** (Phase 2+):

1. Decide: is the opcode an extension of an existing dispatch slot, or a new one?
2. If new: pick an unused handler number in range `0x4D-0x5C` (currently free) and
   add a dispatch entry in the `H_4A → H_4B → ... → H_4C` chain.
3. Implement the emitter as `L(H(NEW)); ...; L(RET()); B();`
4. Run `node create-mini-kyc3.js` and inspect `projects/mini-kyc.ky` to verify the
   new lines are present.
5. Run `bash scripts/bootstrap-check.sh --strict --lock`. Expect baseline drift — review.
6. If intentional, `bash scripts/bootstrap-check.sh --strict --update-baseline`.

**Do not skip steps 4-5**. Skipping is how flash v4 happened.

## 11. Host Compiler Structure (`ky-compiler.js`)

```javascript
function parse(text)    { /* .ky text → [{op, args}] */ }
function analyze(tokens) { /* group into handlers, strings, blobs, top-level */ }

function compile(src) {
  // 1. Build empty PE to get ptrMap (IAT addresses)
  const pe = new PE();
  pe.setCode(Buffer.alloc(TEXT_VS, 0x90));
  pe.setData(Buffer.alloc(1, 0));
  pe.build();
  const P = pe.ptrMap;  // IAT addresses, used by ci()

  // 2. Build empty PE AGAIN to get final IAT addresses (after code.tell())
  const peFix = new PE();  // ... same setup
  peFix.setCode(Buffer.alloc(code.tell(), 0x90));
  peFix.build();

  // 3. Emit code, then patch IAT fixups using peFix.ptrMap
  // 4. Build final PE with pe.setCode(code), pe.setData(data)
  return pe.build();
}
```

The two-pass IAT fixup dance is because the `.text` section size changes the `.rdata`
section's RVA, which changes IAT addresses. Don't try to "simplify" it.

### ci() — RIP-relative IAT call

```javascript
function ci(name) {
  const r = P[name];                    // first-pass IAT address
  if (r === undefined) return;
  const c = 0x1000 + code.tell();       // current RVA (before emit)
  E.call_rip(code, r - (c + 6));        // emit FF 15 <disp32>
  code.iatFixups.push({
    n: name,
    dispPos: code.tell() - 4,
    instrEnd: c + 6                     // post-emit RIP
  });
}
```

The first-pass `r - (c + 6)` is overwritten by the IAT fixup loop:

```javascript
const cr = peFix.ptrMap[f.n];           // second-pass IAT address
code.b.writeInt32LE(cr - f.instrEnd, f.dispPos);
```

`f.instrEnd` was computed at first-pass time, so the relocation is correct even if the
code size changed.

## 12. Bootstrap Gate (Stage 1)

```bash
# Quick check (no baseline, just self-consistency)
bash scripts/bootstrap-check.sh

# Strict (writes bootstrap-report.txt and -diff.txt)
bash scripts/bootstrap-check.sh --strict

# Lock mode (CI gate — exit 4 if drift detected)
bash scripts/bootstrap-check.sh --strict --lock

# Accept current as new baseline (after intentional change)
bash scripts/bootstrap-check.sh --strict --update-baseline
```

What it does:
1. Runs `node create-mini-kyc3.js` twice. Hashes both `projects/mini-kyc.ky` outputs.
   Must be equal.
2. Runs `node ky-compiler.js projects/mini-kyc.ky <out>` twice. Hashes both.
   Must be equal.
3. If `--strict`: writes `bootstrap-report.txt` with current SHA256.
4. If `--lock`: compares report against `bootstrap-baseline.txt`. Exits non-zero on drift.
5. If `--update-baseline`: overwrites baseline with current SHA256.

**If `bootstrap-check.sh` doesn't exist**: it was deleted by mistake. Restore from
`babb051` (see commit `babb051c5d0828a6046edf364db9c5c96792f035`):

## 13. Development Workflow

### Adding a new opcode emitter (correct procedure)

```bash
# 1. Edit create-mini-kyc3.js to add the emitter
#    - find an unused H_xx slot
#    - add dispatch entry
#    - add handler implementation

# 2. Regenerate and inspect
node create-mini-kyc3.js
diff <(git show HEAD:projects/mini-kyc.ky) projects/mini-kyc.ky | head -50
#    - confirm only the new lines appear
#    - confirm no other lines changed

# 3. Compile and check determinism
node ky-compiler.js projects/mini-kyc.ky /tmp/new.exe
node ky-compiler.js projects/mini-kyc.ky /tmp/new2.exe
cmp /tmp/new.exe /tmp/new2.exe && echo "determinism OK"
sha256sum /tmp/new.exe /tmp/new2.exe

# 4. Run gate
bash scripts/bootstrap-check.sh --strict --lock
#    - if drift: is it just the new emitter in output? expected.
#    - if unexpected drift elsewhere: STOP, debug

# 5. If intentional, accept baseline
bash scripts/bootstrap-check.sh --strict --update-baseline

# 6. Commit (small, single-purpose)
git add create-mini-kyc3.js projects/mini-kyc.ky bootstrap-baseline.txt
git commit -m "feat: emit opcode 0xXX (description)"
```

### Debugging a Stage 2 crash

When `mini-kyc.exe` crashes on `input.ky`:

```bash
# 1. Confirm it's reproducible
./mini-kyc.exe < input.ky    # crash?
./mini-kyc.exe < /dev/null    # EXIT 0 (template only)?

# 2. Try the smallest input that uses the failing opcode
#    (e.g. projects/test-phase1.ky)

# 3. Compare the output.exe against a known-good reference
node ky-compiler.js input.ky /tmp/ref.exe
./mini-kyc.exe < input.ky    # produces output.exe
cmp output.exe /tmp/ref.exe   # should be byte-identical for simple inputs

# 4. If mismatch, hex-diff to find which handler emitted wrong bytes
xxd /tmp/ref.exe > /tmp/ref.hex
```

### Common pitfalls (lessons from past disasters)

1. **Don't change `create-mini-kyc3.js` without running the gate.** flash v4 added
   ~2000 lines across 16 new opcode emitters + 3 architectural changes, all without
   ever running `bootstrap-check.sh`. Result: no way to know which change broke what.

2. **Don't change multiple things at once.** Each Phase 2 emitter addition should be
   its own commit. Architecture changes (startup blob, EOF rename, WriteFile size)
   are separate commits. If you mix them, bisect becomes useless.

3. **Stage 1 first, Phase 2 second, Phase 3 last.** The TASKS.md plan says this
   explicitly. The order exists because each stage is a prerequisite for the next.

4. **Generator output must be deterministic.** Don't introduce anything that depends
   on `Date.now()`, `Math.random()`, object key order (use `Object.keys` explicitly),
   etc. The bootstrap-check.sh will catch it.

5. **Host compiler output must be deterministic too.** Same rules apply inside
   `ky-compiler.js`. The `analyze()` function uses `Object.keys(prog.handlers)` —
   the order of these matters for label positions.

6. **IAT disp constants** — the formula is
   `constant = 0x4000 + func_idx*8 - 4`,
   `disp32 = constant - state_0E_at_computation`.
   Re-derive before Phase 2, don't blindly trust `.flash-v4-create-mini-kyc3.patch`.

7. **The `H_62` overload bug** — see §9. Fix it before Stage 2, not after.

8. **The embedded PE template is 0x8800 bytes, not the full 0x14400.** The `84` opcode
   in `create-mini-kyc3.js:133` copies only the first 0x8800 bytes. Stage 1 outputs
   fit, but Stage 2 with full opcodes may not. Verify before relying on `84`.

## 15. When You're Stuck

If something is confusing and the spec doesn't cover it:

1. Read the file you're confused about end-to-end. These files are small (<900 lines).
2. Run the code and observe. Add `console.log` to `ky-compiler.js` if needed.
3. Diff against `git show babb051:create-mini-kyc3.js` — the pre-H_30 version is
   simpler and may illuminate the current structure.
4. Check `git log -- <file>` for context on recent changes.
5. Don't change anything until you understand the existing code. **Adding code is easy,
   understanding is hard. Reverse the priority.**

If you find a bug or invariant violation, fix it in a small, isolated commit.
**Never bundle "while I'm here" changes with bug fixes.**

If you make a mess: `git stash`, get back to a clean state, restart with smaller steps.
The flash-v4 stash (`FLASH-V4-REVIEW.md`) is the cautionary tale.
